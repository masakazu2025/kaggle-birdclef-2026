## 1. 学習パイプライン全体像

### 全体フロー

```
mel画像（縦×横の2次元）
        ↓
EfficientNet（w約1980万個）← 特徴を抽出
        ↓
Global Average Pooling（GAP）← 空間方向を平均で潰す
        ↓
1280次元の特徴ベクトル
        ↓
Classifier head（w約20万個）← 特徴を206種のlogitに変換
        ↓
Sigmoid ← logitを0〜1の確率に変換
        ↓
206種の確率ベクトル　← 予測
        ↓
Loss関数 ← 外れ具合を数値化
        ↓
backprop ← wへの影響を計算（自動微分）
        ↓
Optimizer ← wを更新
```

### 【予測】と【学習】の分離

```
【予測】← 推論時も学習時も同じ処理
mel画像 → EfficientNet → Classifier head → Sigmoid → 206種の確率ベクトル

【学習】← 訓練時だけ追加で走る
予測と正解を比較 → Loss関数 → backprop → Optimizer
```

PyTorchが担う範囲：**mel画像生成（librosaなど）より後、つまりEfficientNetへの入力以降を全部**。mel画像への変換はPyTorchの外（前処理）。

## 2. Mel spectrogram

### 定義

音声を2次元画像に変換する手法。

- 横軸：時間
- 縦軸：周波数
- 色の濃さ：音の強さ（dB）

### 変換プロセス

```
音声波形（1次元）
    ↓ 短時間フーリエ変換（STFT）
    ↓ 窓をスライドさせながら周波数成分を取り出す
スペクトログラム（時間×線形周波数）
    ↓ melフィルタバンクをかける
    ↓ 低周波を細かく、高周波を粗く集約
mel spectrogram（時間×mel周波数）
    ↓ dBスケール変換（対数変換）
最終的な画像
```

### melスケールの数式

mel周波数は次のように定義される：

```
mel(f) = 2595 × log₁₀(1 + f / 700)
```

fはHz単位の周波数。低周波域ではmelとfがほぼ線形に対応し、高周波域では対数的に圧縮される。

### なぜmelスケールを使うのか

**人間の耳のためではない**。これは重要な点。

真の理由は「限られたピクセル数で有効な特徴量を最大化する」ための工夫。

- 線形スケールだと低周波側（鳥の鳴き声の基音帯域）の情報が粗くなる
- 高周波側に無駄なマスが割り当てられる
- melスケールで低周波を細かく、高周波を粗く分割することで、識別に役立つ情報を効率よく詰め込める

### メリット

- 計算コストが画像と比べて軽い（GPUで高速に処理できる）
- 既存の画像認識モデル（EfficientNet等）をそのまま流用できる
- 時間×周波数という2軸で特徴が見やすい

### デメリット

- 位相情報が失われる（振幅のみを扱う）
- パラメータ（窓長、ホップ長、mel bin数）の選択が結果に影響する
- 元の音声に戻せない

### なぜ音声をそのまま使わないのか

EfficientNetは「2次元配列のパターンを認識する仕組み」。1次元の波形データをそのまま食わせても機能しない。2次元に変換するためにmel spectrogramが必要。

## 3. Augmentation

### 定義

訓練データを加工してサンプルを水増しする手法。

### 主な手法

| 手法 | 内容 | 効果 |
|------|------|------|
| ノイズ付加 | ホワイトノイズや環境音を混ぜる | 実環境の録音条件に強くなる |
| Time masking | 時間軸の一部をマスク（0埋め） | 一部の情報が欠けても予測できるようになる |
| Frequency masking | 周波数軸の一部をマスク | 特定帯域への依存を減らす |
| Mixup | 2つのサンプルを重み付きで混合 | 決定境界が滑らかになる、汎化性能向上 |
| Pitch shift | 音程を少しずらす | 個体差への耐性 |
| Time stretch | 再生速度を変える | 鳴き方の速さの違いに対応 |

### メリット

- データセットを実質的に拡張できる（少ない元データでも学習が進む）
- 過学習を防ぐ
- 実環境の多様な状況に対応できるようになる

### デメリット

- 過度にかけると学習が難しくなる
- 手法の選択・強度の設定が結果に影響する（ハイパーパラメータが増える）

### Dropoutとの関係

目的の起点が違う：

```
Augmentation　← 入力データの多様性を増やす
                「様々な音声環境に対応できるモデルを作る」

Dropout　　　　← モデル内部の依存性を減らす
                「特定の特徴への過依存を防ぐ」
```

ただし結果として両方とも「汎化性能を上げる」という方向に働くので、計算結果として近くなる。「その情報に特化させない」という意図でAugmentationを使うなら、目的が重なる。

## 4. チャンク分割

### 目的

**モデルに食わせられる形にすること**。

モデルは固定サイズの入力しか受け取れないため、長さがバラバラな音声を5秒ごとに切り揃える。

### 設計パラメータ

| パラメータ | 内容 | 典型値 |
|------|------|------|
| チャンク長 | 1チャンクの時間長 | 5秒 |
| stride | 次のチャンクの開始位置までの間隔 | 5秒（重複なし）または2.5秒（50%重複） |
| overlap | 重複部分の長さ | stride < チャンク長 なら重複あり |

### 5秒である理由

- BirdCLEFの公式評価が5秒単位で行われる
- 鳥の鳴き声の多くが5秒以内に収まる
- 短すぎると文脈が失われ、長すぎると1チャンクに複数種が含まれやすくなる

### 注意点

- 予測精度向上は副次的な効果で、**主目的は入力サイズの統一**
- 学習時にランダムな開始位置からチャンクを切り出すと、データ多様性が増える（実質的なAugmentation）

## 5. EfficientNet

### 本質的な位置づけ

**「2次元配列のパターンを認識する仕組み」。画像に限らない。**

### 人間の解釈とモデルの視点

「エッジ」「テクスチャ」は人間の解釈。モデルからすれば数値の変化パターンの階層的な集約。

```
低層：隣接する数値の差　　　　　← 人間が「エッジ」と呼ぶもの
中層：差のパターンの繰り返し　　← 人間が「テクスチャ」と呼ぶもの
高層：パターンの組み合わせ　　　← 人間が「オオルリっぽい音」と呼ぶもの
```

抽象度が上がるにつれて、人間が名前をつけているだけ。モデルの中では一貫して**数値の変化パターンの階層的な集約**が起きているだけ。

### mel spectrogramでの解釈例

```
周波数軸方向の急激な変化 → 音の立ち上がり
時間軸方向の周期的パターン → 鳴き声のリズム
特定の周波数帯の強度分布 → 音色の特徴
```

これらも全部、モデルからすれば数値の変化パターン。

### EfficientNet-V2系列

| モデル | パラメータ数 | 最終出力次元 | 用途 |
|------|------|------|------|
| V2-S（Small） | 約2,000万 | 1280 | 軽量で高速、BirdCLEFで定番 |
| V2-M（Medium） | 約5,400万 | 1280 | バランス型 |
| V2-L（Large） | 約1億2,000万 | 1280 | 精度最優先 |

B0〜B7はEfficientNetの旧系列。V2は2021年にリリースされた改良版。

### Global Average Pooling（GAP）

畳み込み層の出力は4次元テンソル：`(batch, 1280, H, W)`

これをH×W方向で平均して潰す：

```
GAP後： (batch, 1280)
```

### GAPのメリット

- 画像サイズが変わっても1280次元に統一できる
- パラメータを持たない（過学習の原因にならない）
- 空間全体の特徴を均等に反映する

### メリット（EfficientNet全体）

- ImageNetで学習済みの高品質なwを活用できる
- パラメータ数と精度のバランスが優れている
- PyTorch/timmで簡単に利用できる

### デメリット

- 音声特有の設計ではない（画像用の設計を流用している）
- 固定サイズの入力を前提とする

## 6. Transfer learning（転移学習）

### 定義

ImageNetで学習済みのwをそのまま持ってきて、BirdCLEFデータでbackpropしながら調整していく。「良い初期値から始める」というイメージ。

### なぜ有効か

ImageNetで2000万枚以上の画像で学習したwは「画像からパターンを抽出する能力」という点で非常に信頼できる。mel spectrogramも2次元画像なので、その検出能力がそのまま使える。

### 2つのアプローチ

```
Feature extraction：
  EfficientNetのwを固定
  Classifier headのwだけ学習
  軽量、データが少ないときに有効

Fine-tuning：
  EfficientNet本体のwも更新する
  データがある程度ある場合に精度が出やすい
```

### 段階的な学習戦略

```
Phase 1：Classifier headだけ学習（EfficientNetは固定）
         ↓
         「この特徴がオオルリに対応する」という対応表が固まる
         ↓
Phase 2：EfficientNetも少しずつ解放して学習
         ↓
         「BirdCLEF的に重要なパターン」に微調整される
```

最初からEfficientNetも全部更新すると、学習済みの良いwが壊れるリスクがある。

### 学習率の使い分け

```
EfficientNet本体　← 学習率を小さく（例：1e-4）
                    すでに良いwなので少しだけ調整

Classifier head　← 学習率を大きく（例：1e-3）
                    ゼロから学ぶので大きく動かす
```

### メリット

- ゼロから学習するより少ないデータと学習時間で良い結果が出る
- 計算リソースが限られていても有効
- 汎化性能が高くなりやすい

### デメリット

- 元のドメインと大きく異なるデータには効きにくい
- Fine-tuningのやり方を誤るとpretrainedの恩恵が失われる

## 7. Classifier head

### 位置づけ

**設計概念**。「EfficientNetの出力を最終予測に変換する部分」という概念で、具体的な中身はLinear、ReLU、Dropout、Sigmoidなどの組み合わせ。

### 役割の比喩

```
EfficientNet本体　← 食材を全部切って、味付けして、盛り付け直前まで仕上げる
Classifier head　← 皿に盛るだけ
```

EfficientNetの1280次元はすでに「良い特徴」が凝縮されているので、それを信頼してClassifier headはシンプルに保つ。

### 行列の中身

```
        オオルリ  ツバメ  メジロ  …（206種）
特徴1  [ 0.3,    -0.2,   0.8,   … ]
特徴2  [ -0.1,    0.9,   0.2,   … ]
特徴3  [ 0.7,    0.1,   -0.3,   … ]
…
特徴1280
```

各位置の意味：「その特徴量がその種にどれだけ関連するか」を表す重み。backpropで学習される。

### 計算式

```
オオルリのlogit = Σ(特徴i × w_オオルリ,i)  for i = 1..1280
               = 特徴1×0.3 + 特徴2×(-0.1) + 特徴3×0.7 + …
```

行列表記：

```
logit = W × feature + b
```

- W: 206×1280の重み行列
- feature: 1280次元ベクトル
- b: 206次元のバイアス
- logit: 206次元のベクトル

### wはランダムな初期値から始まる

Classifier headはBirdCLEF専用の部分なのでImageNetの学習済みwが存在しない。ゼロから学習。

### 構成例

| 構成 | 内容 | 用途 |
|------|------|------|
| シンプル | `Linear(1280→206) → Sigmoid` | EfficientNetを信頼する標準設計 |
| 中間層あり | `Linear(1280→512) → ReLU → Dropout → Linear(512→206) → Sigmoid` | 特徴をもう一段抽象化 |
| 安定重視 | `Linear(1280→512) → BatchNorm → ReLU → Dropout → Linear(512→206) → Sigmoid` | BatchNormで学習を安定化 |

### シンプルな構成のメリット

- 過学習リスクが低い
- 学習が速い
- パラメータが少ない（約26万個）

### シンプルな構成のデメリット

- 表現力が限定される
- 複雑なタスクでは精度が頭打ちになる可能性

### 中間層を入れるメリット／デメリット

```
メリット：
- 表現力が上がる
- 非線形性が増える

デメリット：
- パラメータが増える
- 過学習リスクが上がる
- 学習率やdropout率の調整が必要
```

## 8. Sigmoid

### 定義

任意の実数（logit）を0〜1に変換する非線形関数。

### 数式

```
σ(x) = 1 / (1 + e^(-x))
```

### 性質

| logit | 確率 | 意味 |
|------|------|------|
| -∞ | 0 | 完全に「いない」 |
| -4 | 約0.018 | 強く「いない」 |
| 0 | 0.5 | 五分五分 |
| +4 | 約0.982 | 強く「いる」 |
| +∞ | 1 | 完全に「いる」 |

### 導関数

```
σ'(x) = σ(x) × (1 - σ(x))
```

これは重要な性質で、順伝播で計算した値をそのまま逆伝播に使えるため実装が効率的。

### wの正負との関係

logitが0のとき確率0.5なので、wは正負どちらも取る：

```
特徴ベクトル × w = logit
        ↓
logit < 0 → Sigmoid → 0.5未満 → 「いない」寄り
logit > 0 → Sigmoid → 0.5超　 → 「いる」寄り
```

### Softmaxとの違い

```
Sigmoid：
  - 各クラス独立に0〜1に変換
  - 合計は1にならない
  - 複数クラスが同時に高い確率を持てる
  - BirdCLEFのような多ラベル分類に適している

Softmax：
  - 全クラス合計が1になるよう正規化
  - 排他的（どれか1つが正解）
  - 単一ラベル分類に適している
```

### メリット

- 解釈しやすい（確率として扱える）
- 微分が計算しやすい
- 多ラベル分類に自然に対応する

### デメリット

- 勾配消失問題：端（0や1）に近づくほど傾きが0に近づき、backpropで勾配がほぼ0になる
- 中間層で使うと深いネットワークで学習が進まなくなる
- そのためEfficientNet本体の中間層ではSigmoidではなくReLUを使う

## 9. ReLU

### 定義

負の値を0に捨て、正の値はそのまま通す非線形関数。

### 数式

```
ReLU(x) = max(0, x)
```

### 導関数

```
ReLU'(x) = 1  (x > 0)
         = 0  (x ≤ 0)
```

### 役割

- 「この特徴は今回の判断に関係ない」という選別
- 関係ある特徴だけを次の層に流す

### 線形変換とReLUの組み合わせが主流な理由

```
Linear → ReLU → Linear → ReLU → …
```

この組み合わせでどんな複雑な関数も近似できる（**普遍近似定理**）。

**メリット**：
- 計算が速い（比較とmaxだけ）
- 微分が単純（0か1）
- 勾配消失が起きにくい
- 実装が安定する

**デメリット**：
- 「Dying ReLU」問題：入力が常に負の値だと、そのニューロンが永遠に更新されなくなる
- 出力の平均が0にならない

### 改良版

| 関数 | 数式 | 特徴 |
|------|------|------|
| Leaky ReLU | `max(0.01x, x)` | 負でも小さい勾配を残す、Dying ReLU対策 |
| GELU | `x × Φ(x)` | Transformer系で主流 |
| Swish/SiLU | `x × σ(x)` | EfficientNetV2で採用 |

EfficientNetV2内部ではSiLUが使われている。

### なぜSigmoidではなくReLUを中間層で使うか

```
線形変換だけを何層重ねても、数学的には1つの線形変換と等価
→ 非線形性が必須

非線形関数の中でも：
- Sigmoidは勾配消失が起きやすい
- ReLUは計算コストが低く、勾配消失が起きにくい
```

## 10. Loss関数（BCE）

### 役割

**予測がどれだけ外れているかを数値化する（採点者）**。

Optimizerの役割と混同しないこと：
- Loss関数 → 採点者
- Optimizer → どう練習し直すかを決めるコーチ

### BCEの思想

**ベルヌーイ分布を仮定**：「鳴いているか、いないか」という二値の事象を独立したコイン投げとして扱う。

- 各クラスを独立に扱う（206クラスの相関は無視する簡略化）
- 「正解データが最もよく説明できる確率分布をモデルに学ばせる」という**情報理論的な発想**

### 確率分布の仮定

BCEが仮定しているのは**ベルヌーイ分布**：

```
P(y | p) = p^y × (1-p)^(1-y)
```

- y: 正解ラベル（0 or 1）
- p: モデルが出力する確率（0〜1）

モデルの出力p=0.3は「このコインの表が出る確率はp=0.3です」という宣言であり、**p自体がベルヌーイ分布のパラメータそのもの**。

### 尤もらしさ（likelihood）

「このデータは、このモデルの仮定からどれくらい自然に生まれそうか」という度合い。

例：p=0.7のコインを3回投げて「表・表・裏」が出た場合の尤もらしさ：

```
0.7 × 0.7 × 0.3 = 0.147
```

BCEは**「正解データが最も自然に生まれてくるようなpをモデルに学ばせる」**ようにする。

### BCE計算式

```
正解y=1のとき：Loss = -log(p)
正解y=0のとき：Loss = -log(1-p)

一般形：
Loss = -[y × log(p) + (1-y) × log(1-p)]
```

### なぜ対数を使うか

```
予測0.9 → -log(0.9) ≈ 0.105 → 小さいLoss
予測0.5 → -log(0.5) ≈ 0.693 → 中くらい
予測0.1 → -log(0.1) ≈ 2.303 → 大きいLoss
予測0.01 → -log(0.01) ≈ 4.605 → 非常に大きい
```

**「自信を持って間違えた」ほど急激に罰する**。単純な差では表現できない「確信度」の概念を組み込める。

### 206クラスのLossの合計

```
全体Loss = オオルリのLoss + ツバメのLoss + メジロのLoss + …（和）
         = Σ BCE(y_i, p_i)  for i = 1..206
```

**積ではなく和**：積にすると1種でも確率が低ければ全体が極小になって意味をなさない。対数を取ることで、確率の積→対数の和に変換できる。

### 個々のLossと合計のLoss

```
個々のLoss　← どの種の予測が外れているかを診断するために見る
合計のLoss　← モデル全体の学習の進捗を監視するために見る
```

backpropでwを更新するときは合計のLossを使う。ただ種ごとのLossを監視することで「全体のLossは下がっているが稀な種の予測が改善していない」という状況を発見できる。

### クラスweightによる補正

BirdCLEFはクラス不均衡（種ごとのサンプル数の偏り）が大きい。

```
pos_weight = 鳴いていないサンプル数 / 鳴いているサンプル数
```

これをBCEに組み込むと：

```
Loss = -[pos_weight × y × log(p) + (1-y) × log(1-p)]
```

PyTorch：`BCEWithLogitsLoss`の`pos_weight`引数。

### 実践：PyTorchでの実装

```python
import torch.nn as nn

# pos_weightは各クラスごとに設定
pos_weight = torch.tensor([...])  # shape: (206,)
criterion = nn.BCEWithLogitsLoss(pos_weight=pos_weight)

# logits: (batch, 206), targets: (batch, 206)
loss = criterion(logits, targets)
```

### メリット

- 多ラベル分類に自然に対応する
- 確率論的に解釈できる
- 計算が効率的
- pos_weightで不均衡に対処できる

### デメリット

- クラス間の相関を無視している
- 極端な不均衡では補正してもまだ難しい
- 全クラスを等しく扱うため、特定クラスの誤分類コストを反映しにくい

### 類似のLoss：Focal Loss

```
FL(p) = -α × (1-p)^γ × log(p)  (y=1のとき)
```

- γ：簡単なサンプルのLossを抑える（難しいサンプルに集中させる）
- α：クラス重み

BirdCLEFで「自信を持って外した」サンプルをより重視したい場合に有効。

## 11. backprop（誤差逆伝播）

### 核心：連鎖律（chain rule）

合成関数の微分は、各関数の微分の積になる：

```
y = f(g(x)) のとき
dy/dx = dy/dg × dg/dx
```

### モデル全体での連鎖

Lossからwまでは関数の連鎖：

```
L → Sigmoid → logit → Classifier head → EfficientNet → w
```

偏微分は：

```
∂L/∂w = ∂L/∂(Sigmoid) × ∂(Sigmoid)/∂(logit) × ∂(logit)/∂(ClsHead出力)
      × ∂(ClsHead出力)/∂(EffNet出力) × ∂(EffNet出力)/∂w
```

この連鎖を**逆向きに辿りながら**、各層での微分を順番に掛け合わせていく。

### 具体例（Classifier head部分）

```
L = -log(σ(Wx + b))  ただしy=1のとき

∂L/∂W = ∂L/∂σ × ∂σ/∂(logit) × ∂(logit)/∂W
      = -(1/σ) × σ(1-σ) × x
      = -(1-σ) × x
      = (σ - 1) × x  ※y=1なら予測と正解の差そのもの
```

**予測と正解の差がそのまま勾配になる**という美しい性質がある。

### PyTorchでの実装

```python
loss.backward()  # これだけで全パラメータの勾配を自動計算
```

PyTorchは計算グラフを自動で構築し、`backward()`で逆向きに辿って全wの勾配を計算する（**自動微分**）。

### 計算の流れ

```
順伝播（予測）：
入力 → 層1 → 層2 → … → 層N → Loss
（各層で計算と中間値の保存）

逆伝播（勾配計算）：
Loss → 層N → … → 層2 → 層1
（連鎖律で各wの勾配を累積）
```

### メリット

- 深いネットワークでも効率的に勾配計算できる
- PyTorchの自動微分でエンジニアは数式を書かなくて良い
- 数千万のパラメータでも現実的な時間で計算できる

### デメリット

- 深いネットワークでは勾配消失・勾配爆発が起きやすい
- 中間値をメモリに保持するため、大きなモデルではメモリ消費が大きい
- Sigmoidなど特定の活性化関数では勾配が消えやすい

## 12. Optimizer

### 役割

backpropで求めた勾配を使ってwを更新する（**コーチ**）。

### 基本式（SGD）

```
w_新 = w_旧 - learning_rate × ∂L/∂w
```

| 勾配 | 意味 | 更新方向 |
|------|------|------|
| ∂L/∂w > 0 | Lossがwと同じ方向に増える | wを減らす |
| ∂L/∂w < 0 | Lossがwと逆方向に増える | wを増やす |
| ∂L/∂w = 0 | Lossがwに影響されない | 動かない |

### learning rate（学習率）

wをどれだけ動かすかを決める値。

```
大きい learning rate：
  メリット：学習が速い、局所解から抜け出しやすい
  デメリット：不安定、発散リスク

小さい learning rate：
  メリット：安定、精度が出やすい
  デメリット：学習が遅い、局所解にはまりやすい
```

### SGD（Stochastic Gradient Descent）

```
w ← w - η × ∇L
```

- η: learning rate
- ∇L: 勾配

**メリット**：
- シンプル
- 計算量が少ない
- メモリ消費が少ない

**デメリット**：
- learning rateの調整が難しい
- パラメータごとに最適な学習率が違う
- 収束が遅い

### Adam

モメンタムとRMSPropを組み合わせた手法：

```
m_t = β₁ × m_{t-1} + (1-β₁) × ∇L    （勾配の移動平均）
v_t = β₂ × v_{t-1} + (1-β₂) × ∇L²   （勾配の二乗の移動平均）

w ← w - η × m_t / (√v_t + ε)
```

- m_t: 一次モーメント（慣性）
- v_t: 二次モーメント（スケール）

**直感**：
- よく更新されるパラメータは学習率を小さく
- あまり更新されないパラメータは学習率を大きく

### AdamW

Adamにweight decayを適切に組み込んだもの：

```
w ← w - η × (m_t / (√v_t + ε) + λ × w)
```

Adamでは正則化（weight decay）の扱いに問題があり、AdamWはそれを解決した。

**メリット**：
- パラメータごとに学習率が自動調整される
- 収束が速い
- 正則化が適切に効く
- 深層学習の多くで標準的

**デメリット**：
- 計算量がSGDより多い
- メモリ消費が多い（各パラメータに m_t, v_t を持つ）

### Learning rate scheduler

学習中に学習率を変化させる仕組み：

| 方式 | 内容 |
|------|------|
| Step decay | 一定epochごとに学習率を減らす |
| Cosine annealing | コサインカーブで徐々に減らす |
| Warmup | 最初は小さく、徐々に上げてから減らす |
| ReduceLROnPlateau | Lossが下がらなくなったら減らす |

BirdCLEFではCosine annealing with warmupがよく使われる。

## 13. 正則化

### 目的：過学習を防ぐこと

過学習とは「訓練データに特化しすぎて、見たことないデータに対応できない状態」。

```
訓練データ：Lossが小さい、よく当たる
検証データ：Lossが大きい、全然当たらない
```

### 過学習の原因と対策

原因は複数あり、それぞれ対策が異なる：

```
原因①：特定のニューロンへの依存　← Dropoutで対策
原因②：特定のwが突出して大きい　← Weight decayで対策
```

原因①の場合、個々のwは小さくても、**いくつかのwの組み合わせに強く依存している**状況も想定される。Weight decayでは検出できないケース。

ただ両方の原因が重なることもあるので、両方同時に使うことが多い。どちらか片方でもある程度は効く。

### 13.1 Dropout

学習中にランダムでニューロンを一定割合で無効化する。

```
通常：　特徴1 → 特徴2 → 特徴3 → 特徴4 → 出力
Dropout：特徴1 → （消）→ 特徴3 → （消）→ 出力
```

毎回ランダムに消えるニューロンが変わる。

### 数式

```
y = x × mask / (1-p)  # 学習時
```

- mask: 各要素が確率(1-p)で1、確率pで0のベクトル
- pは無効化する割合（例：0.2なら20%のニューロンを無効化）
- `/(1-p)` は期待値を保つためのスケーリング

### なぜ過学習を防げるか

- 特定のニューロンに依存した予測ができなくなる
- 「オオルリはこの特徴1だけで判断できる」という暗記ができなくなり、他の特徴でも判断できるよう学習せざるを得なくなる
- 1つの事象を複数の角度から見て、複数の観点で予測可能にする、**多角的な視点を作ることが目的**

### 現実との対応

実際の野外音声では：
- 背景ノイズで一部の特徴が消える
- 別の鳥の鳴き声と重なって特徴が隠れる
- 録音機材の質によって特徴が欠ける

Dropoutはその状況を学習中に意図的に再現している。

### 推論時の扱い

```
学習時：ニューロンをランダムに消す
推論時：全ニューロンを使う（消さない）
```

PyTorchでは`model.train()`と`model.eval()`の切り替えがこれに対応。

### アンサンブルとしての解釈

Dropoutは統計的に言うと**複数の独立した予測器のアンサンブル**を1つのモデルで実現している。毎回異なるニューロンの組み合わせで学習することで、様々な「視点」を持つ予測器が内部に形成される。

### メリット

- 実装が簡単
- 学習時のコストのみで推論時は軽い
- 過学習を効果的に防げる
- アンサンブル効果で汎化性能が上がる

### デメリット

- Dropout率の調整が必要
- 学習に時間がかかる（収束が遅くなる）
- BatchNormと組み合わせるときに注意が必要

### 13.2 Weight decay

wの値が大きくなりすぎることにペナルティを与える。

### 数式

Loss関数にペナルティ項を追加：

```
L_total = L_original + λ × ||w||²
         = L_original + λ × Σ w_i²
```

この形をL2正則化と呼ぶ。

### 更新式への影響

```
通常のwの更新式：
w_新 = w_旧 - learning_rate × 勾配

Weight decay追加：
w_新 = w_旧 - learning_rate × 勾配 - learning_rate × λ × w_旧
     = (1 - learning_rate × λ) × w_旧 - learning_rate × 勾配
```

λが正則化の強さを決めるハイパーパラメータ。wが大きいほどペナルティが大きくなるので、自然とwが小さく保たれる。

### なぜwが大きいと過学習になるか

wが大きいということは、特定の特徴に対して「強く反応しすぎる」状態：

```
wが大きい
  ← 少しの入力変化で予測が大きく変わる
  ← 訓練データの細かいノイズまで拾ってしまう
  ← 過学習
```

### L1正則化との違い

```
L2正則化（weight decay）：
  ペナルティ = λ × Σ w_i²
  wを小さくする（0にはならない）

L1正則化：
  ペナルティ = λ × Σ |w_i|
  wを0にする（スパースになる）
```

深層学習ではL2が主流。

### メリット

- 実装が簡単
- 推論コストに影響しない
- 汎化性能が上がる
- AdamWなどに組み込まれている

### デメリット

- λの調整が必要
- 適切に設定しないと学習が進まなくなる
- 全パラメータを一律に制約する（層ごとの区別がない）

### DropoutとWeight decayの比較

```
Dropout　　　← まんべんなく使わせる（依存の分散）
Weight decay　← 突出を抑える（値のバランス）
```

両方を組み合わせることが多い。ただし両方同時にフルで入れるとどちらが効いているか分からなくなるので、まず片方から試すアプローチも合理的。

### その他の正則化手法

| 手法 | 内容 |
|------|------|
| Early stopping | validation lossが下がらなくなったら学習を止める |
| Label smoothing | 正解ラベルを1.0ではなく0.9などにする |
| Data augmentation | 入力データを加工して多様性を増やす |
| Mixup | 2つのサンプルを混ぜる |
| Stochastic depth | 層単位でランダムにスキップ |

## 14. 評価指標（cmAP・ROC-AUC）

### 14.1 AP（Average Precision）

### 定義

1クラスに対する評価指標。モデルが出した確率の高い順にサンプルを並べたとき、上位に正解がどれだけ集まっているかを測る。

### 計算例

正解が全部で3件ある場合：

```
1位: 確率0.95 → 正解　← Precision = 1/1 = 1.00
2位: 確率0.90 → 正解　← Precision = 2/2 = 1.00
3位: 確率0.85 → 不正解
4位: 確率0.80 → 正解　← Precision = 3/4 = 0.75
```

正解が出てきたタイミングのPrecisionだけ記録：

```
AP = (1.00 + 1.00 + 0.75) / 3 = 0.917
```

### 数式

```
AP = Σ P(k) × ΔR(k)
```

- P(k): 上位k件でのPrecision
- ΔR(k): k位で正解が出たときのRecallの増分

### Precisionの定義

```
Precision@k = 上位k件中の正解数 / k
```

### 直感

**上位に不正解が混じるほどAPが下がる**。モデルが「自信を持って予測したものが本当に正解か」を測る。

### 14.2 cmAP（classwise mean Average Precision）

### 定義

各クラスごとに独立してAPを計算し、平均を取る：

```
cmAP = (オオルリのAP + ツバメのAP + … + 206種目のAP) / 206
```

### 通常のmAPとの違い

```
通常のmAP（物体検出などで使う場合）：
全クラス・全サンプルを混ぜてランキングを作ってAPを計算

cmAP：
クラスごとに独立してランキングを作ってAPを計算し、平均を取る
```

通常の形だとサンプルが多い種が支配的になる。cmAPはクラスごとに独立しているので、稀な種も平等に扱われる。

### BirdCLEFでの重要性

BirdCLEFのような「種ごとにサンプル数が極端に偏る」データでは：

- サンプルが多い種で高精度を出しても、cmAPはあまり上がらない
- 稀な種の予測精度を上げることがスコアに直結する
- **1種1票**として平等に扱われる

### メリット

- クラス不均衡に強い
- 稀なクラスの性能を正しく評価できる
- ランキングの質を測れる

### デメリット

- 閾値を直接反映しない（確率のランキングのみ）
- 各クラスに最低1つの正解サンプルが必要
- サンプル数が極端に少ないクラスでは不安定

### 14.3 ROC-AUC

### ROC曲線

モデルの予測確率に対して閾値を0〜1の間で動かしたとき、TPRとFPRがどう変化するかを曲線で表したもの。

```
TPR（True Positive Rate）= 実際に「いる」サンプルのうち、正しく「いる」と判定できた割合
FPR（False Positive Rate）= 実際に「いない」サンプルのうち、誤って「いる」と判定した割合
```

### 数式

```
TPR = TP / (TP + FN) = TP / 実際にいる総数
FPR = FP / (FP + TN) = FP / 実際にいない総数
```

| 略 | 意味 |
|------|------|
| TP | True Positive（正しく「いる」と判定） |
| FP | False Positive（誤って「いる」と判定） |
| TN | True Negative（正しく「いない」と判定） |
| FN | False Negative（誤って「いない」と判定） |

### 閾値を動かしたときの変化

```
閾値を下げる（例：0.5 → 0.2）
  「いる」と判定するハードルが下がる
  → より多くのサンプルを「いる」と判定
  → TPRが上がる（正解をより多く拾える）
  → FPRも上がる（不正解も誤って拾う）

閾値を上げる（例：0.5 → 0.8）
  「いる」と判定するハードルが上がる
  → より少ないサンプルしか「いる」と判定しない
  → TPRが下がる（取りこぼしが増える）
  → FPRも下がる（誤って拾う数が減る）
```

### 分母が閾値に依存しない重要性

```
TPR → 分母は「実際にいる」サンプル数（固定）
FPR → 分母は「実際にいない」サンプル数（固定）
```

分母が閾値に依存しないので、閾値を動かしても安定して比較できる。

### AUC（Area Under the Curve）

ROC曲線の下の面積：

```
AUC = 1.0　← 完璧な予測（曲線が左上の角に張り付く）
AUC = 0.5　← ランダムな予測（対角線）
AUC = 0.0　← 完全に逆の予測
```

### ROC曲線の形

```
TPR
1.0 |　　＿＿＿
    |　／
    |／
0.0 |＿＿＿＿＿
    0.0　　　1.0  FPR
```

曲線が**左上に張り出すほど**AUCが大きくなる。「FPRを低く保ちながらTPRを高くできる」ほど左上に張り出す。

### cmAPとROC-AUCの違い

```
cmAP　　← 上位にどれだけ正解が集まっているかを測る（ランキングの質）
ROC-AUC　← 閾値を変えたとき全体的にどれだけ正解と不正解を分離できているかを測る（分離能力）
```

### メリット

**cmAP**：
- 稀なクラスも平等に評価
- ランキングの質を直接評価
- BirdCLEF 2026の公式指標

**ROC-AUC**：
- 閾値に依存しない
- 解釈が直感的
- クラス不均衡にもある程度強い

### デメリット

**cmAP**：
- 計算がやや複雑
- サンプル数が極端に少ないクラスで不安定

**ROC-AUC**：
- 極端なクラス不均衡では楽観的な値が出やすい
- 閾値を直接最適化しない

## 15. Threshold

### 定義

モデルの出力確率を「いる／いない」の二値に変換する境界値。

```
Threshold = 0.5（デフォルト）のとき：
確率 > 0.5 → 「いる」
確率 ≤ 0.5 → 「いない」
```

### なぜThresholdの設定が重要か

デフォルトの0.5が最適とは限らない。

BirdCLEFのような稀な種が多いデータでは：

```
稀な種は学習データが少ないため、モデルが自信を持って高い確率を出しにくい
  ↓
確率0.3でも実際には「いる」ことが多い
  ↓
Threshold=0.5だと取りこぼしてしまう
  ↓
その種のThresholdを0.3に下げるべき
```

### 種ごとにThresholdを最適化する

```
オオルリ　← Threshold = 0.4
ツバメ　　← Threshold = 0.5
メジロ　　← Threshold = 0.3
…
```

検証データで種ごとにAPが最大になるThresholdを探して設定する。

### 最適化の手順

```
1. validation dataで各種の予測確率を取得
2. 各種について、Thresholdを0.01〜0.99まで動かす
3. 各Thresholdでの評価指標（F1やAP）を計算
4. 最大になるThresholdを採用
```

### メリット

- 推論時の計算コストが増えない
- 大きな精度向上が期待できる
- 種ごとの特性に対応できる

### デメリット

- validation dataへの過学習リスク
- 設定がデータセットに依存する
- cmAPのようなランキング指標には直接影響しない

## 16. TTA（Test Time Augmentation）

### 定義

推論時にAugmentationを適用する手法。

### 通常の推論との違い

```
通常の推論：
音声チャンク1つ → モデル → 確率ベクトル
1つの入力に対して1つの予測

TTAの推論：
音声チャンク（元）　　　　　→ モデル → 確率ベクトル①
音声チャンク（ノイズ付加）　→ モデル → 確率ベクトル②
音声チャンク（時間シフト）　→ モデル → 確率ベクトル③
        ↓
①②③の平均 → 最終的な確率ベクトル
```

同じ音声を複数の加工バージョンで予測して、その平均を取る。

### 数式

```
final_prediction = (1/N) × Σ model(augment_i(x))
```

- N: TTAのバリエーション数
- augment_i: i番目のAugmentation

### なぜ精度が上がるか

1つの予測にはモデルの「ブレ」がある。複数の視点から予測して平均を取ることで、そのブレが打ち消し合う。

Dropoutの思想と似ていて、「**多角的な視点を組み合わせる**」という考え方。

### 統計的な解釈

```
個別の予測 = 真の値 + ノイズ
複数予測の平均 = 真の値 + (ノイズの平均)
                        ↓
                     ノイズが打ち消し合って小さくなる
```

### 典型的なTTA手法

| 手法 | 内容 |
|------|------|
| Horizontal flip | 左右反転（画像の場合） |
| Time shift | 時間軸のシフト |
| Pitch shift | ピッチを少し変える |
| Noise addition | 軽いノイズを加える |
| Multi-crop | 複数の位置からクロップ |
| Multi-scale | 異なるサイズで推論 |

### トレードオフ

```
TTAのバリエーションを増やす：
  メリット：精度が上がる
  デメリット：推論時間が増える

例：
  TTA 1倍（通常）    → 推論時間 1x
  TTA 5倍           → 推論時間 5x、精度+1〜3%
  TTA 10倍          → 推論時間 10x、精度+2〜4%
```

### BirdCLEFでの注意点

BirdCLEFは**推論時間に厳しい制限**がある：

- 約60GB分の音声データを2時間以内に処理
- TTAを増やしすぎるとタイムアウト
- 効果の高いTTAを少数選ぶのが鉄則

### メリット

- モデルを変えずに精度向上できる
- アンサンブルの一種として機能
- 実装が比較的簡単

### デメリット

- 推論時間が増える
- Augmentationの選択が重要（悪いAugmentationは精度を下げる）
- 学習時のAugmentationと整合性を取る必要がある

## 17. Validation戦略とleakage

### cross-validation

### 仕組み

```
データ全体を5分割（fold）する

fold1をvalidation、残り4つで学習 → スコア1
fold2をvalidation、残り4つで学習 → スコア2
fold3をvalidation、残り4つで学習 → スコア3
fold4をvalidation、残り4つで学習 → スコア4
fold5をvalidation、残り4つで学習 → スコア5
        ↓
5回の平均 → 最終的な検証スコア
```

### メリット

- データを有効活用できる
- 評価が安定する
- 過学習の検出がしやすい

### デメリット

- 学習コストが5倍
- foldの分割方法に注意が必要

### leakage（データ漏洩）

学習データとvalidationデータに、本来分離すべき情報が混在してしまう状態。

### BirdCLEFで起きがちなleakage

```
音声ファイルAを5秒ごとに分割：
チャンク1 → 学習データ
チャンク2 → validationデータ  ← 同じファイルから！
チャンク3 → 学習データ
```

同じファイルのチャンクは：
- 録音環境が同じ
- 同じ個体の鳴き声が含まれる
- 背景ノイズが似ている

モデルが「validationデータを実質的に学習している」状態になる。

### 対策

```
チャンク単位ではなく、ファイル単位でfoldを分割する

ファイルAのチャンク全部 → fold1
ファイルBのチャンク全部 → fold2
…
```

### その他のleakageパターン

| パターン | 内容 | 対策 |
|------|------|------|
| File leakage | 同じファイルから分割 | ファイル単位で分割 |
| Time leakage | 近い時間帯のデータが混在 | 時間で分割 |
| Location leakage | 同じ録音地点のデータが混在 | 地点で分割 |
| Individual leakage | 同じ個体の鳴き声が混在 | 個体ID情報があれば個体で分割 |

### StratifiedKFold

クラス分布を保ちながら分割する手法：

```
通常のKFold：
fold1に稀な種が全く含まれない可能性

StratifiedKFold：
各foldで種の分布比率が保たれる
```

BirdCLEFのようなクラス不均衡データで重要。

### Multi-label StratifiedKFold

多ラベル分類（1サンプルに複数ラベル）では、`iterative-stratification`ライブラリの`MultilabelStratifiedKFold`を使うのが一般的。

### 検証スコアと本番スコアの乖離

```
CV score（手元での検証）= 0.85
LB score（コンペのリーダーボード）= 0.78
差 = 0.07
```

この差が大きい場合、leakageが起きている可能性が高い。

### 実践的な対策

```
1. ファイル単位でfold分割
2. StratifiedKFoldで分布を保つ
3. CV scoreとLB scoreの相関をチェック
4. 乖離が大きいときはfold分割を見直す
```

### メリット（適切なValidation戦略）

- 本番性能の正確な予測
- 過学習の早期発見
- モデル選択の判断材料

### デメリット

- 設計に時間がかかる
- ドメイン知識が必要
- データによって最適な戦略が変わる

## 付録A：学習ループの疑似コード

```python
# 疑似コード
for epoch in range(num_epochs):
    model.train()
    for batch in train_loader:
        mel, label = batch
        
        # 予測
        feature = efficientnet(mel)       # (batch, 1280)
        logit = classifier_head(feature)  # (batch, 206)
        prob = sigmoid(logit)             # (batch, 206)
        
        # Loss計算
        loss = BCE(prob, label)
        
        # 勾配計算
        optimizer.zero_grad()
        loss.backward()                   # 自動微分
        
        # wの更新
        optimizer.step()
    
    # Validation
    model.eval()
    with torch.no_grad():
        val_score = evaluate(model, val_loader)  # cmAP
    
    # Learning rate調整
    scheduler.step()
    
    # Early stopping判定
    if early_stopping(val_score):
        break
```
